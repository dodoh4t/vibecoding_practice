# dodoTodoList PRD

## 1. 개요 (목적 & 배경)

### 목적

dodoTodoList는 개인 사용자가 웹 브라우저에서 자신의 할 일을 간단하게 등록, 조회, 완료 처리, 삭제할 수 있도록 제공하는 TodoList 웹 앱이다.

### 배경

개인 개발자나 1인 사용자는 복잡한 협업 기능보다 빠르게 할 일을 기록하고 처리 상태를 관리할 수 있는 단순한 도구가 필요하다. dodoTodoList는 회원가입과 로그인 기반의 개인 Todo 관리 기능에 집중하며, Supabase PostgreSQL을 데이터베이스로 사용하되 인증은 백엔드 자체 JWT 방식으로 처리한다.

### 제품 범위 요약

- 프론트엔드: React + Vite + TypeScript
- 백엔드: Node.js + Express + JavaScript + pg
- 데이터베이스: Supabase PostgreSQL
- 인증: 백엔드 자체 JWT 인증
- 배포: Vercel

## 2. 타깃 사용자

### 주요 사용자

- 혼자 할 일을 관리하고 싶은 개인 사용자
- 개인 프로젝트나 학습, 일상 업무를 정리하려는 1인 개발자

### 사용자 특성

- 빠른 Todo 등록과 완료 처리를 선호한다.
- 팀 협업, 알림, 복잡한 프로젝트 관리 기능은 필요하지 않다.
- 웹 브라우저 기반으로 데스크톱과 모바일 화면 모두에서 접근하기를 원한다.

## 3. 사용자 스토리

### 3.1 회원가입

나는 개인 사용자로서 내 Todo를 안전하게 관리하기 위해 회원가입을 원한다.

**Acceptance Criteria**

- 이메일과 비밀번호를 입력해 계정을 생성할 수 있다.
- 이메일은 유효한 이메일 형식이어야 한다.
- 비밀번호는 최소 8자 이상이어야 한다.
- 이미 가입된 이메일로는 회원가입할 수 없다.
- 회원가입 성공 후 사용자는 로그인 화면에서 생성한 계정으로 로그인할 수 있다.

### 3.2 로그인

나는 개인 사용자로서 내 Todo 목록에 접근하기 위해 로그인을 원한다.

**Acceptance Criteria**

- 가입한 이메일과 비밀번호로 로그인할 수 있다.
- 로그인 성공 시 백엔드에서 발급한 JWT로 인증 상태가 유지된다.
- 로그인 성공 후 사용자는 Todo 목록 화면으로 이동한다.
- 잘못된 로그인 정보 입력 시 명확한 오류 메시지가 표시된다.

### 3.3 할 일 추가

나는 개인 사용자로서 해야 할 일을 기록하기 위해 할 일 추가를 원한다.

**Acceptance Criteria**

- 로그인한 사용자는 Todo 내용을 입력해 새 Todo를 등록할 수 있다.
- 빈 내용의 Todo는 등록할 수 없다.
- 앞뒤 공백만 있는 Todo는 등록할 수 없다.
- 등록된 Todo는 기본적으로 미완료 상태로 저장된다.
- Todo 등록 성공 후 입력 필드는 비워지고 목록에 새 Todo가 표시된다.

### 3.4 완료 체크

나는 개인 사용자로서 작업 진행 상태를 관리하기 위해 완료 체크를 원한다.

**Acceptance Criteria**

- 로그인한 사용자는 자신의 Todo를 완료 또는 미완료 상태로 변경할 수 있다.
- 상태 변경 결과는 새로고침 후에도 유지된다.
- 다른 사용자의 Todo 상태는 변경할 수 없다.

### 3.5 할 일 삭제

나는 개인 사용자로서 불필요한 항목을 정리하기 위해 할 일 삭제를 원한다.

**Acceptance Criteria**

- 로그인한 사용자는 자신의 Todo를 삭제할 수 있다.
- 삭제된 Todo는 목록에서 더 이상 보이지 않는다.
- 삭제 실패 시 기존 Todo 목록 상태는 유지된다.
- 다른 사용자의 Todo는 삭제할 수 없다.

### 3.6 전체 목록 조회

나는 개인 사용자로서 내 작업을 한눈에 확인하기 위해 전체 목록 조회를 원한다.

**Acceptance Criteria**

- 로그인한 사용자는 자신이 등록한 모든 Todo를 볼 수 있다.
- 각 Todo는 내용과 완료 여부를 함께 표시한다.
- 로그인하지 않은 사용자는 Todo 목록에 접근할 수 없다.

- 낮음: 현재 PRD 범위에서는 계획하지 않으며, 필요 시 후속 버전에서 검토한다.

## 5. 비기능 요구사항

### 5.1 반응형 웹 UI

- 최소 지원 해상도는 360px x 640px이며, 390px x 844px, 768px x 1024px, 1024px x 768px, 1280px x 720px 이상 화면에서 주요 흐름이 정상 동작해야 한다.
- TailwindCSS의 기본 브레이크포인트인 `sm` 640px, `md` 768px, `lg` 1024px, `xl` 1280px 기준으로 레이아웃이 자연스럽게 전환되어야 한다.
- 모바일에서는 Todo 입력, 목록, 액션 버튼이 세로 흐름으로 배치되어야 한다.
- 태블릿과 데스크톱에서는 입력 영역과 목록 영역이 더 넓은 화면을 활용하도록 정렬되어야 하며, 주요 작업 버튼은 한 화면에서 쉽게 찾을 수 있어야 한다.
- 모든 지원 해상도에서 텍스트, 입력 필드, 버튼, Todo 항목이 서로 겹치거나 화면 밖으로 잘리지 않아야 한다.

### 5.2 성능

- 초기 화면은 불필요한 대용량 리소스 없이 빠르게 로드되어야 하며, 일반적인 4G 또는 가정용 네트워크 환경에서 첫 화면 렌더링까지 3초 이내를 목표로 한다.
- Todo 목록 조회, 추가, 완료 체크, 삭제 요청은 일반적인 네트워크 환경에서 1초 이내의 사용자 피드백을 제공해야 한다.
- 프론트엔드는 Vite 기반 빌드 결과물을 Vercel에서 정적 자산으로 배포한다.
- 백엔드는 Vercel 환경에서 Express API로 동작하며, DB 연결은 `pg` 라이브러리를 사용한다.

### 5.3 보안

- Supabase Auth는 사용하지 않으며, 백엔드 자체 JWT 인증을 사용한다.
- 비밀번호는 평문으로 저장하지 않고 안전한 해시 알고리즘을 사용해 저장해야 한다.
- JWT 서명 키는 환경 변수로 관리해야 하며 클라이언트 코드에 노출되어서는 안 된다.
- JWT는 만료 시간을 가져야 하며, 만료되었거나 위조된 토큰은 거부해야 한다.
- 인증이 필요한 Todo API는 유효한 JWT가 없으면 접근할 수 없어야 한다.
- 사용자는 자신의 Todo에만 접근, 수정, 삭제할 수 있어야 한다.
- SQL 쿼리는 `pg`의 파라미터 바인딩을 사용해 SQL Injection을 방지해야 한다.

### 5.4 접근성

- 주요 버튼과 입력 필드는 키보드만으로 조작할 수 있어야 한다.
- 입력 필드에는 명확한 레이블 또는 접근 가능한 이름이 제공되어야 한다.
- 완료 체크박스, 삭제 버튼 등 주요 인터랙션 요소는 스크린 리더에서 의미를 이해할 수 있어야 한다.
- 색상만으로 완료 상태나 오류 상태를 전달하지 않아야 한다.

### 5.5 사용성

- 사용자는 로그인 후 한 화면에서 Todo 입력과 전체 목록 확인을 할 수 있어야 한다.
- Todo 추가, 완료 체크, 삭제 후 목록 상태가 즉시 갱신되어야 한다.
- 오류 발생 시 사용자가 다음 행동을 이해할 수 있는 메시지를 표시해야 한다.

### 5.6 호환성 및 배포

- 최신 Chrome, Edge, Safari, Firefox 브라우저에서 정상 동작해야 한다.
- 프론트엔드와 백엔드는 Vercel에 배포 가능해야 한다.
- 데이터베이스는 Supabase PostgreSQL의 DB 기능만 사용하며, Supabase Auth, Storage, Realtime 등은 사용하지 않는다.

## 8. 데이터 모델 요구사항

- 사용자 계정은 `users` 테이블에 저장하며, `id`, `email`, `password_hash`, `created_at`, `updated_at` 컬럼을 가진다.
- `users.email`은 필수 값이며, 이메일 형식 검증과 중복 방지를 적용해야 한다.
- `users.email`은 대소문자 중복 가입을 방지하기 위해 소문자로 정규화해 저장한다.
- `users.password_hash`에는 평문 비밀번호가 아니라 안전한 해시 값만 저장한다.
- Todo 항목은 `todos` 테이블에 저장하며, `id`, `user_id`, `content`, `is_completed`, `created_at`, `updated_at` 컬럼을 가진다.
- `todos.user_id`는 `users.id`를 참조하는 외래 키이며, 사용자가 자신의 Todo만 조회, 수정, 삭제하도록 검증하는 기준이다.
- `todos.content`는 필수 값이며, 빈 문자열이나 공백만 있는 값은 저장할 수 없어야 한다.
- `todos.is_completed`는 필수 값이며, 새 Todo는 기본적으로 `false`로 저장한다.
- 한 명의 사용자는 여러 Todo를 가질 수 있고, 하나의 Todo는 반드시 한 명의 사용자에게만 속한다.
- JWT를 무상태 토큰으로 운영하는 MVP 범위에서는 별도 `sessions` 테이블을 필수로 두지 않는다.
